# AI_RULES.md — AI 開発ルール

このリポジトリで開発するすべての AI（Claude Code / ChatGPT / Cursor / Copilot 等）が守るルール。
人間の開発者にも同じルールが適用される。

## 大原則

1. **既存コードを必ず読んでから書く。** 類似機能（例: 新しいリスト系ページなら `ShoppingPage`、モーダルなら既存モーダル実装）を最低 1 つ読み、そのパターンを踏襲する。ゼロから独自流で書かない
2. **依頼されたことだけをやる。** ついでのリファクタリング・リネーム・整形・依存追加をしない。改善案がある場合は実装せず提案として報告する
3. **一度に変更しすぎない。** 1 つの変更セット（PR）は 1 つの目的。無関係なファイルに触れない
4. **壊すより遅い方がまし。** 確信が持てない破壊的変更（テーブル削除、RLS 変更、認証フロー変更）は必ず人間に確認する

## コーディングルール

- **言語は JavaScript (JSX)。** TypeScript への変換・`.ts`/`.tsx` ファイルの新規追加はしない（Edge Functions 配下 `supabase/functions/` のみ Deno/TypeScript）。TS 移行はプロジェクトオーナーが明示的に決定した場合のみ
- **型安全の担保**: TS がない分、Supabase の `select()` は必要なカラムを明示し、`?.` と `??` で null 安全に書く。複雑な構造には JSDoc を付けてよい（必須ではない）
- **DRY は 3 回目から**: 同じコードが 3 箇所目に現れる時に共通化する。早すぎる抽象化をしない
- **KISS**: ライブラリ追加より標準 API・既存依存で解決する。依存追加は理由を添えて提案し、承認を得てから
- **ファイルを肥大化させない**: 新規ページは 500 行以内を目安にし、超えそうならコンポーネント分割する。既存の巨大ページ（`PlacesPage` 1200 行超等）は「これ以上大きくしない」— 触るついでの分割はしないが、そのページに機能追加する場合は新規部分を別コンポーネントに切り出す
- **エラー処理を必ず書く**: Supabase 呼び出しは `error` を確認し、失敗時はユーザーに日本語で分かるフィードバック（もしくは楽観的更新のロールバック）を行う。`console.error('〜エラー:', error)` 形式でログも残す
- **コメントは最小限**: 「なぜ」だけを書く。「何をしているか」のコメント・変更経緯のコメントは書かない。既存コード同様、コメントは日本語
- **UI 文言・aria-label は日本語**
- **CSS はデザイントークンを使う**: 色・影・角丸のハードコード禁止。`var(--primary)` 等 `src/index.css` のトークンを使う（[DESIGN.md](./DESIGN.md)）。ダークモードはトークンが自動対応するため、`prefers-color-scheme` をページ側で書く必要は原則ない

## データベースのルール

- スキーマ変更は必ず `supabase/migrations/NNN_名前.sql` を**追加**する（既存ファイルの書き換え禁止）
- 新テーブルには必ず: `family_id` 外部キー（家族スコープのデータの場合）/ `ENABLE ROW LEVEL SECURITY` / `get_my_family_id()` を使った RLS ポリシー、をセットで書く
- リアルタイム同期が必要なら `ALTER PUBLICATION supabase_realtime ADD TABLE 〜` も同じマイグレーションに含める
- マイグレーションは自動適用されない。**PR / 完了報告に「適用が必要なマイグレーション」を明記する**
- 詳細は [DATABASE.md](./DATABASE.md)

## AI 開発フロー（必須）

いきなりコードを書かない。以下の順で進め、1〜4 は着手前に簡潔に報告する。

1. **要件整理** — 依頼を自分の言葉で言い直し、曖昧な点は確認する（自明なら省略可）
2. **現状分析** — 関連する既存コード・テーブル・ドキュメントを読む
3. **実装方針の提案** — どのファイルをどう変えるか、新規テーブル / マイグレーションの要否
4. **影響範囲の確認** — 他ページ・RLS・Realtime・PWA キャッシュへの影響
5. **実装** — 上記方針に沿って最小の差分で
6. **自己レビュー** — diff を読み直し、デバッグ用コード・未使用 import・不要ファイルを除去
7. **リファクタリング** — 今回の変更内で明らかな重複があれば整理（スコープ外に広げない）
8. **検証** — `npm run build` 成功を確認。UI 変更は該当画面の動作を説明できること
9. **完了報告** — 変更内容・適用が必要なマイグレーション・確認してほしい点を報告

軽微な修正（typo、1 行のスタイル調整等）は 1〜4 を省略してよい。

## 完了条件（Definition of Done）

以下をすべて満たした場合のみ「完了」と報告してよい。

- [ ] `npm run build` が成功する
- [ ] 変更に関係する画面がライト / ダーク両モードで崩れない（トークンを使っていれば通常自動で満たされる）
- [ ] スマホ幅（375px 目安）でレイアウトがはみ出さない
- [ ] エラー時のハンドリング（error 確認・ユーザーフィードバック）がある
- [ ] 未使用 import・デバッグ用 `console.log`・コメントアウトされた死にコードがない
- [ ] 新テーブルに RLS ポリシーがある / 必要なマイグレーション SQL が追加されている
- [ ] `aria-label`（アイコンのみのボタン）等、既存水準のアクセシビリティを維持している
- [ ] 明らかな N+1 クエリや全件フェッチなど、性能を悪化させる実装がない
- [ ] [CHANGELOG.md](../CHANGELOG.md) の `Unreleased` に変更を 1 行追記した

> **注**: 本プロジェクトには現状テストランナー・Linter がない。導入されるまでは `npm run build` がビルドゲート。「テスト成功」「Lint エラーなし」は導入後に有効化する。

## やってはいけないこと（明示的な依頼がない限り）

- TypeScript 化、状態管理ライブラリ導入、CSS フレームワーク導入などの技術スタック変更
- 既存マイグレーションファイルの編集・削除
- RLS ポリシーの緩和（`USING (true)` 化など）
- `main` ブランチへの直接 push
- 依存パッケージの追加・メジャーバージョンアップ
- 既存機能の削除・URL パスの変更
- `.env` の値の出力・コミット
